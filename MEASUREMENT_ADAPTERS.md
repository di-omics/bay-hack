# On-site measurement adapters

The scientific loop already accepts a physical value through
`PlrMcpBench(measurement_fn=...)`. Choose the fastest venue lane below. Do not
rewrite the controller.

## Evidence ladder

| Label | Meaning |
|---|---|
| `modeled` | generated by the fallback assay simulator |
| `measured:reader-csv` | read from a plate-reader export and calibrated to 0 to 1 |
| `measured:camera` | computed from a real plate image using in-frame references |
| `hardware-validated` | measured value also cleared the real volume and CV gates |

A measurement is not hardware-validated merely because it came from a device.

## Lane 1: plate-reader CSV

Export one row per destination well:

```csv
well,rfu
B1,602
C1,884
D1,731
```

Check one value using low and high RFU references:

```bash
python -m bayhack.measurements csv run_artifacts/reader.csv B1 \
  --raw-low 100 --raw-high 1100
```

Wire the same file into the physical bench:

```python
from bayhack.loop import DBTLLoop
from bayhack.measurements import CsvWellMeasurement, LinearSignalCalibration
from bayhack.seams import PlrMcpBench

measurement = CsvWellMeasurement(
    "run_artifacts/reader.csv",
    calibration=LinearSignalCalibration(raw_low=100, raw_high=1100),
)
bench = PlrMcpBench(backend="VENUE_BACKEND", measurement_fn=measurement)
loop = DBTLLoop(bench, target_signal=0.85)
```

The adapter rereads the CSV for every well, so a venue export can be updated
between rounds without restarting the process.

## Lane 2: camera colorimetry

Use a fixed top-down camera. Keep exposure, white balance, distance, and lighting
constant. In the fallback plate map, A1 is the high-color stock and A2 is the
low-color diluent. They remain in every image as in-frame references.

Create a calibration from the plate's pixel bounding box:

```bash
python -m bayhack.measurements calibrate \
  --box 120,80,1080,720 \
  --low-well A2 --high-well A1 \
  --output run_artifacts/camera-calibration.json
```

Capture each round as `plate_B1.jpg`, `plate_C1.jpg`, and so on. Check a well:

```bash
python -m bayhack.measurements camera \
  run_artifacts/plate_B1.jpg \
  run_artifacts/camera-calibration.json B1
```

Render the sampled regions over the image before trusting the geometry:

```bash
python -m bayhack.measurements overlay \
  run_artifacts/plate_B1.jpg \
  run_artifacts/camera-calibration.json \
  run_artifacts/camera-overlay.png
```

The camera evidence records the image path and SHA-256 digest, target-well RGB, A2 low-reference
RGB, A1 high-reference RGB, plate geometry, and normalized value. Reader CSV
evidence records the source path and digest, raw RFU, calibration, and normalized value.

Wire the image pattern into the bench:

```python
from bayhack.loop import DBTLLoop
from bayhack.measurements import CameraWellMeasurement
from bayhack.seams import PlrMcpBench

measurement = CameraWellMeasurement(
    "run_artifacts/plate_{well}.jpg",
    "run_artifacts/camera-calibration.json",
)
bench = PlrMcpBench(backend="VENUE_BACKEND", measurement_fn=measurement)
loop = DBTLLoop(bench, target_signal=0.85)
```

Camera support imports Pillow lazily:

```bash
pip install pillow
```

## Physical validation

Do not set a hardware-validation label by hand. Use the shipped measured gates:

```python
from bayhack.verification import CsvVolumeGate, JsonCvCheckpoint

bench = PlrMcpBench(
    backend="VENUE_BACKEND",
    measurement_fn=measurement,
)
loop = DBTLLoop(
    bench,
    rhodamine_fn=CsvVolumeGate("run_artifacts/volume-gate.csv"),
    cv_fn=JsonCvCheckpoint("run_artifacts/cv_{well}_{run_id}.json"),
    target_signal=0.85,
)
```

The loop earns `hardware-validated` only after both measured gates pass. See
[VERIFICATION_ADAPTERS.md](VERIFICATION_ADAPTERS.md) for exact file formats,
criteria, fail-closed behavior, and the zero-motion preflight command.

## Dry-run checklist

1. `python -m bayhack.preflight` reports `SIMULATION_READY` and zero motion.
2. `python -m bayhack.safety` shows zero robot commands for an invalid plan.
3. The adapter CLI returns a finite value between 0 and 1.
4. The ledger says `measured:reader-csv` or `measured:camera`.
5. A failed physical gate leaves `world_model.updated` false.
6. ACCEPT requires both the 0.85 objective threshold and uncertainty clearance.
7. The follow-up transfer to H12 uses the same verified backend path.

## Stage-safe replay

Once the hardware run has produced a receipt, present it without issuing another
robot command:

```bash
python -m bayhack.dashboard --receipt run_artifacts/trust.json
```

The dashboard reports `safe receipt replay` and preserves the original
measurement provenance, RGB or RFU evidence, physical gates, objective result,
uncertainty, and follow-up action.
